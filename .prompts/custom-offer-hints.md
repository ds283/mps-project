## PREAMBLE

This task is designed to implement a new feature, where after a Go-Live event that initiates a selection period,
or when a new LiveProject is added during the selection cycle, the convenor is shown "hints" prompting them
to set up CustomOffers for students who have had previous supervision contact with a faculty member.

This is a large refactoring, so write a file to disk keeping track of your progress. Update this file as you
proceed with the task.

## TASK 1

During the Go Live Celery workflow, for each SelectingStudent that is inserted into the database, determine whether
the student has any previous SubmittingStudent instances that have already been retired. We wish to keep track
of the faculty members who own projects assigned to any SubmissionRecord instances for these retired
SubmittingStudent instances. These should be persisted in the database using a new "CustomOfferHint" model.

The model should track the SubmissionRecord that has been used to generate the hint.

We only wish to generate CustomOfferHints where the faculty member is offering projects in to be selected in
the project class that is going live.

This task can be done asynchronously. It can be set up as a separate Celery workflow that proceeds in the background.
The main Go Live task does not need to wait for it to complete (but we should track whether it has completed).

## TASK 2

On the convenor dashboard for "Student selections are currently live" we want to display any available hints in a
compact table similar to the "Popular projects" table (generated by a "dashboard_tile" macro).
In each row, show the student's name, the academic year of the previous contact, the project that was previously
assigned, and the faculty owner.

The table should offer "Reject" and "Accept" buttons. The "Reject" button should simply delete the CustomOfferHint
instance.
The "Accept" button should generate CustomOffer instances for all LiveProjects owned by the faculty member
that are associated with the current project class.

The user interface needs to be compact so that it does not take up too much vertical space on the page.
Place it just above the "Popular projects" table. Keep the same visual style as the "Popular projects" table
so that the user interace looks coordinate and has a visual unity.

The `comment` field of the CustomOffer should note that the offer has been generated from an accepted hint,
and note the previous supervision year and previous supervision project.

You should also generate StudentJournalEntry instances for each of the created CustomOffers that record that the
offer was generated by accepting a hint. Record the name of the user who accepted the hint, the date/time, the
previous supervision year, the previous supervision project. Record the project and supervisor that the
CustomOffer has been generated for.

# TASK 3

LiveProject instances can be added after a GoLive event has completed using the convenor.inject_liveproject() route.
If the project class is still in its selection phase, then after each injection we want to search through all
attached SelectingStudent members and decide whether CustomOfferHints should be generated for them, using the
same criteria described in TASK 1.

We want to share as much code as possible between TASK 1 and TASK 3. Do not repeat yourself if it can be avoided.
