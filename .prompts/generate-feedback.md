## PREAMBLE

This task is part of a broader complex of refactoring tasks aimed at rebuilding the legacy feedback system based on
SubmissionPeriodRecord into a new, workflow-centered system based on MarkingEvent and MarkingWorkflow and their
associated models.

In a previous task, the Celery workflows to produce feedback documents were refactored to work with the new system.
These workflows are in @app/tasks/marking.py. The main ones are `generate_feedback_reports()`
and `generate_feedback_report`. `generate_feedback_report()` generates a `FeedbackOrchestrationJob` instance
and activates the orchestration step coordinator `global_feedback_orchestration_step`.

What is now required is to wire these tasks into the user interface via the `MarkingEvent` inspector.

General implementation patterns:

- Views that provide lists (of `FeedbackAsset`, `FeedbackTemplate`, `FeedbackRecipe`, etc.) should use a Datatables
  based front end backed by an AJAX endpoint. Implement an AJAX row formatted in @app/ajax/convenor.
- Use the `ServerSideSQLHandler` pattern where possible to provide sorting, searching, and pagination in the AJAX
  endpoint. Use `ServerSideInMemoryHandler` where it is not possible to build a single SQL query that will satisfy
  all requirements.
- Do not use separate "display" and "sortstring" fields in Datatables rows when `ServerSideSQLHandler` is used.
  These do not format correctly and are not needed. `ServerSideSQLHandler` will handle and sorting required.
- Use the `select2()` library to render `QuerySelectField` and `QueryMultipleSelectField` fields. Use the
  `select2-small` value for the `selectionCssClasss` and `dropdownCssClass` properties.

Place new Jinja2 templates in app/templates/convenor/feedback.

Place new AJAX formatters in app/ajax/convenor.

This is a large refactoring. Please generate an on-disk progress file and update it as you work, so that state
can easily be recovered if the implementation runs over more than one rate-limit window.

## TASK 1 - Initiate generation of feedback

When a `MarkingEvent` advances to the `READY_TO_GENERATE_FEEDBACK` state, a call-to-action should be generated by the
`MarkingWorkflow.get_convenor_actions` method showing that this option is now available.

The `Actions` dropdown menu for a `MarkingEvent` in the state `READY_TO_GENERATE_FEEDBACK` should gain a
"Generate feedback..." menu item. Place this beneath the "Conflation reports..." menu item described below.
In the `READY_TO_PUSH_FEEDBACK` state, this menu item should instead be labelled "Fill missing feedback...".

For `LLMOrchestrationJob`, the job is built and the orchestration manager is dispatched synchronously, rather than
as part of a Celery workflow. We want to adopt the same approach for `FeedbackOrchestrationJob`. The menu item
described above should link to a route `convenor.generate_marking_event_feedback`, to be placed in
app/convenor/markingevent.py. This logic currently in the `generate_feedback_reports` task should be moved here.
This route should synchronously build the required `FeedbackOrchestrationJob` instance and activate
`global_feedback_orchestration_step` to dispatch the task. `generate_feedback_reports` will then be unused and can be
removed.

The route must offer the user a choice of `FeedbackRecipe` instances associated with this project class, to be
used in generating the feedback. A new field `recipe` will be required on `ConflationReport`. Store the label
of the feedback recipe as a character string. There is no significant value in storing a FK to the
`FeedbackRecipe` record, because this may change over time. The cost/benefit in committing an archived version
of the recipe and all its assets is not realistic, so we should simply keep some record of which recipe was used.

When feedback is generated and emplaced on a `ConflationReport`, the `MarkingWorkflow` should move into the
`READY_TO_PUSH_FEEDBACK` state.

## TASK 2 – A `ConflationReport` inspector

When a `MarkingEvent` advances to the `READY_TO_GENERATE_FEEDBACK` state, it will be linked to a full complement of
`ConflationReport` instances. Generate a "Conflation reports..." menu item that links to an inspector view.
Place this on the "Actions" menu underneath the "Inspect workflows..." item.

The visual style and appearance of the inspector should parallel the `SubmitterReport` inspector. It should include
a context panel at the top summarizing details of the parent `MarkingEvent`. It should include a state summary showing
how many `ConflationReport` instances have successfully generated feedback, and how many are stale.

The inspector should include the following columns:

- "Student", showing the student name and candidate number (same as `SubmitterReport` inspector
- "Project", showing the project name and the project owner's name
- "Conflated grades", showing the conflated targets stored in `ConflationReport.conflation_report`. Include the target
  label and the same formatting to the "Grade" column of the `SubmitterReport` inspector. Show who generated the
  report (`generated_by`) and the generation timestamp.
- "Feedback", showing the titles of any feedback reports stored in `ConflationReport.feedback_reports` together with
  a small thumbnail preview and an indication of the recipe used to produce them. Include an indication of whether
  feedback has been sent, the name of the user who initiated the push, and the timestamp.
- If feedback is currently being generated for this `ConflationReport`, include a visual indication parallel to the
  indicator on the convenor's submitter inspector. See the implementation in
  app/templates/convenor/submitter_macros.html. This will require new fields on `ConflationReport` which should be  
  parallel to those on `SubmissionRecord`
- An "Actions" menu dropdown. This should include:
    - If the `is_stale` flag is set and feedback has not yet been sent, a "Reconflate" option to perform
      re-conflation just for this `ConflationReport`. After success, the `is_stale` flag should be cleared.
    - If one or more feedback reports have been generated, btu `feedback_sent == False`, an option to
      "Delete" the feedback report, and an option to "Regenerate" it. Regeneration should offer the user a choice of
      available templates; the regeneration step need not use the same template that was used initially.
    - If feedback is deleted, and no `ConflationReport` instances have any surviving feedback, then the
      `MarkingWorkflow` should regress to the `READY_TO_GENERATE_FEEDBACK` state.
    - If one or more feedback reports have been generated, but `feedback_sent == False`, a "Send feedback..." option
      to dispatch feedback for just this student. After dispatch (see below), `feedback_sent`, `feedback_push_id`
      and `feedback_push_timestamp` should be set correctly. The dispatched email should be linked into
      `feedback_emails`.
    - If there are linked emails in `feedback_emails`, a link to a view listing these emails with links to the normal
      email log viewer. Ensure that return URLs and suitable text descriptions are threaded through the linking chain.

Further, actions that cause `is_stale` to be set elsewhere in the codebase should be made unavailable (routes gated
with an error message, menu options disabled) is feedback has already been **sent** for this student, i.e.,
`feedback_sent` is `True`. Prefer not to write repeated ad hoc queries for this; define a suitable helper function
if needed to cleanly encapsulate any query logic.

## TASK 3 - Push feedback tasks

The push tasks in @app/tasks/push_feedback.py need to be refactored to work with the new `MarkingEvent` system.

Push events can be initiated for a single `ConflationReport`, as described above, or for a `MarkingEvent` as a whole.
When the `MarkingEvent` is in the `READY_TO_PUSH_FEEDBACK` state, its Actions dropdown menu should offer a
"Send feedback..." option that applies to all `ConflationReport` instances in the `MarkingEvent`. This menu
item should trigger a `push_marking_event_feedback` task as described below.

When a push event is initiated, new `EmailWorkflow` instances must be generated. Generate a **single** instance of
`EmailWorkflow` to encapsulate all student emails generated in the event, and a **single** instance to encapsulate
all faculty emails generated. In the case of push for a single `ConflationReport`, there will be one student email,
but possibly multiple faculty emails.

The push event (whether for a single `ConflationReport` or an entire `MarkingWorkflow`) should use a form to
query for a delay period, which should be used to set the delay time on the `EmailWorkflow` instances.
Default to 1 hour. The form should also offer an option to dispatch emails to a test email address rather
than the target email addresses. It should offer options to dispatch notifications to staff with supervisor roles,
marking roles, and moderation roles.

Student emails are built by `push_student_feedback`. This will need to be refactored to accept a PK for a
`ConflationReport` instance instead of `SubmissionRecord`. The `_collect_feedback_attachments` method can simply be
replaced by the collection of `feedback_reports` available on the `ConflationReport`.

Notice that, currently, this task generates an `EmailWorkflow` per student. As explained above, this needs to be
replaced by ONE `EmailWorkflow` per push event.

You will need to set up a callback belonging to the **student** email to write the `feedback_sent` flag, the timestamp,
and to link the `EmailLog` generated into the `feedback_emails` collection. Faculty email dispatch can fail,
but provided the student email is correctly dispatched, we want to set the `feedback_sent` flag.

Faculty emails are built by `push_role_feedback`. This also needs to be refactored to accept a PK for a
`ConflationReport` instance. You will need to schedule instances of this task for each user with a `SubmissionRole`
associated with the student in the `ConflationReport`. Use the role selections (supervisor/marker/moderator) captured
by the dispatch form to determine which `SubmissionRole` instances to include. Link faculty emails into the
`feedback_emails` collection on `ConflationReport`.

For `push_student_feedback` and `push_role_feedback` equally, the subject payload should include the subject tag,
student name, and project class. The body payload should include the student's `SubmittingStudent` instance as
`sub`, their `SubmissionRecord` instance as `sr`, their `StudentData` instance as `sd`, the project class as `pclass`,
the `SubmissionPeriodRecord` as `period` and the `ConflationReport` as `cr`.

If feedback is being pushed from an entire `MarkingEvent`, `notify_feedback_push` can be used to surface an outcome
to the initiating user. This task will need to be rebuilt to capture correct statistics from the
`push_role_feedback` and `push_student_feedback` workflows separately.

The `push_period` task should be replaced with a `push_marking_event_feedback` task performing the equivalent job for
the set of `ConflationReport` instances attached to a `MarkingEvent`. All email tasks should be idempotent. Do NOT
dispatch email where `ConflationReport.feedback_sent` indicates it has already been sent.
