## PREAMBLE

This task is part of an ongoing refactoring of marking and feedback funcions onto the new MarkingEvent/MarkingWorkflow
framework. In this task, the intention is to build a marking form that allows assessors (the markers belonging to a
MarkingWorkflow) to return their reports.

### Coding conventions

Please read @CLAUDE.md for general coding conventions.

Please remember NOT to generate database migrations. They will be generated by hand.

### TASK 1

Once a MarkingReport moves to the `distributed` state, we wish to allow the owning faculty member to access a web form
to return their report, until the `grade_generaed_by_id` field is set to not None. This shows that the form has
been successfully submitted. The form should remain editable for 1 day, but after that it closes and is not
editable unless this field is cleared by the convenor. The form needs to be discoverable from their dashboard.

Include an option on the MarkingReport inspector for the convenor to clear the `grade_generated_by_id` field
(and likewise the `grade_generated_timestamp` field) so that a form can be returned to the user if needed.

At the moment, display of marker assignments on faculty dasboards is gated by `PeriodClassConfig.display_marker` in
@app/templates/faculty/dashboard/submitter_card.html. The marker assignment is rendered by the marker_assignments()
macro.

When a marking form is available, this must override the `PeriodClassConfig.display_marker` setting. We also need a
call to action on the faculty dashboard showing clearly which marking tasks need attention.

There is a more general problem, because the current dashboard level is compact, but makes this information hard
to discover. Faculty tasks are hidden under the "My Enrolments" tab, which produces a large dropdown "megamenu" rather
than using a large number of nav pills. See the markup in @app/templates/faculty/dashboard/dashboard.html.
Please consider alternatives and recommend a better UI layout.

### TASK 2

Each row in the table laid out by marker_assignments() in @app/templates/faculty/dashboard/assignments/marker.html
represents a single SubmissionRole instance. We need to find all MarkingReport instances that belong to this
SubmissionRole and have `distribution` set to True. There are various status flags in MarkingReport including
`report_submitted`, `signed_off_by`, `signed_off_timestamp`, `grade` and `feedback_submitted` that should be
compactly summarized to the user at this level.

We then need a button clearly linking to the marking form for this report.

### TASK 3

We need to generate a template for the marking form and appropriate backend logic.

The marking form should be accessible to faculty supervisors, convenors, users with "admin" role that share a Tenant,
and users with the "root".

For convenors, and "admin" or "root" users, the form should compactly surface any StudentJournalEntry items that
belong to the same year (represented by MainConfig) as the SubmissionPeriodRecord -> ProjectClassConfig that owns
the parent MarkingEvent. Include a link to the view_journal_entry() route, with appropriate return URL and text,
so that these journal entries can easily be read. Include a short summary of the journal entry on the marking form.

For users wth ROLE_SUPERVISOR or ROLE_RESPONSIBLE_SUPERVISOR, we wish to surface the information recorded in
SupervisionEvent reports. Please design a compact horizontal layout showing clearly the progression of
SupervisionEvent instances associated with this SubmissionRecord. Where the `monitor_attendance` flag is set,
they should clearly highlight (by colour coding?) which events were marked absent (notified or not),
which were present (late or not), and which were rescheduled. Also show events where attendance data was not
recorded in a light grey faded colour. Include an overall attendance figure. Please see the markup in
@app/templates/projecthub/hub.html .

We also wish to indicate to supervisors which events have a meeting summary or private supervision notes, without
taking up too much space. Consider how this information can be presented. Should it use an offcanvas slideover or
similar UI element?

### TASK 4

Now proceed to lay out the report form. This should be based on the `LiveMarkScheme` associated with the
`MarkingWorkflow`.

Please include the `title` and `rubric`, one below the other, as directly-injected HTML elements marked |safe.
The constructor of the mark scheme has already been responsible for the layout of these elements.

The form should give a download link to the student's report. Include a suitable thumbnail preview.
It should also include any attachments linked to the `MarkingWorkflow` via its `attachments` fields. Generate
an attractive but compact layour for these, including thumbnails.

Underneath, please use the `schema` to determine the layout of the form. Use the long comment at the beginning of the
`MarkingScheme` class to understand its structure.

For each block in the "scheme" element, lay out its "title" and "description" in a suitable format.
For the following elemnts in "field", lay out the "text" and a form control for the "field_type". You will need to
dynamically build a WTForms class to handle the form submission. Include suitable controls for each of the
"field" entries, with names determined by the "key" field.

Include appropriate validators and defaults for each type:

- "boolean": respect the "default" field
- "number": validators for the "min", "max" fields. Respect the "precision" field on submission, respect "default"
- "percent": implicit validators min=0, max=100, precision=0 (no decimal places), respect "default"
- "text": respect the "default" field if given

The validator for the JSON schema parse_schema() in @app/shared/forms/wtf_validators.py should be adjusted
to verify that "key" is suitable to be a Python identified.

If the `uses_standard_feedback` flag is set, include TextAreaField elements for the `feedback_positive` feedback, badged
as "What was good?" and the `feedback_negative` feedback, badges "What could be improved next time?".

On submission, write the `feedback_positive` and `feedback_negative` fields to the `MarkingReport` instance, if
`uses_standard_feedback` is set. Generate a `feedback_timestamp`.

Now extract the values of each of the "field" blocks defined in the JSON `schema` from the submitted form. Ensure that
they are cast to objects of the appropriate type:

- "boolean": bool
- "number": float
- "percent": float
- "text": str
  If a ValueError is raised then the form should fail validation.

Produce a dict of the values of the "field" blocks.

Now proceed to the "validation" block. for each block, evaluate the "test" field by regarding it as a Python expression.
It should be evaluated using eva() or a safe alternative using the "field" dict described above. Each test should
evalaute to a boolean expression. If it evaluates to True, the test passes. If it evaluates to False, consider
what to do, as follows.

- "prevent_submit": the form does not validate. Report the "message" from this block as an error message.
- "email": generate an EmailWorkflow and EmailWorkflowItem to each person on the `notify_on_validation_failure`
  colection, MARKING_VALIDATION_FAILURE email template type. Include the `MarkingReport` instance and the dict of
  "field" blocks in the payload.
- "web": take no action immediately, but surface this report to the convenor on the MarkingReport inspector, as
  described below.

Serialize the dict of "field" values into a JSON string and store this in `MarkingReport.report`.
Set the `report_submitted` flag to True. Clear the `signed_off_id` and `signed_off_by` and `signed_off_timestamp`
fields.

Generate the final `grade` value using the Python expression in the "conflation_rule" element of the mark scheme
`schema`. This is a Python expression that should evaluate to a float. Use eval() or a similar safe alternative
to evaluate this expression given the dict of "field" values stored in `report`. Store the result in `grade`.

Store the identity of the current user in `grade_generated_by_ud` and generate a timestamp in
`grade_generated_timestamp`.
