# Task: Implement card-based submitters view

## Background

The convenor submitters view at `app/templates/convenor/dashboard/submitters.html`
currently renders students as rows in a DataTables table, with per-student submission
period detail generated by the `project_tag` macro in
`app/templates/convenor/submitters_macros.html` and served via an AJAX formatter in
`app/ajax/convenor/submitters.py`. The route is `submitters()` in
`app/convenor/submitters.py`.

This task replaces that DataTables view with a server-rendered card layout. The data
layer and route logic are untouched; only the template and rendering path change.

A reference HTML mockup of the target design is at `mockups/submitters_cards_mockup.html`.
Study it carefully before writing any code — it is the authoritative visual and structural
specification for this work. Pay particular attention to the CSS in the mockup's `<style>`
block, which uses Bootstrap 5.3 CSS custom property tokens throughout. The implemented
template must follow the same token-based approach.

---

## Colour policy

All colours in the new template and its `<style>` block must be expressed as Bootstrap 5.3
CSS custom property tokens. The full token vocabulary is documented at
https://getbootstrap.com/docs/5.3/customize/css-variables/. The tokens used in this design
include:

**Structural / neutral:**

- `--bs-body-bg`, `--bs-body-color` — default background and text
- `--bs-secondary-bg`, `--bs-tertiary-bg` — muted background tiers (card headers, filter panels)
- `--bs-secondary-color` — muted text (metadata, labels)
- `--bs-border-color`, `--bs-border-color-translucent` — borders and subtle dividers
- `--bs-border-radius`, `--bs-border-radius-sm`, `--bs-border-radius-lg` — corner radii

**Semantic states:**

- `--bs-primary`, `--bs-primary-text-emphasis`, `--bs-primary-bg-subtle`, `--bs-primary-border-subtle`
- `--bs-success`, `--bs-success-text-emphasis`, `--bs-success-bg-subtle`, `--bs-success-border-subtle`
- `--bs-warning`, `--bs-warning-text-emphasis`, `--bs-warning-bg-subtle`, `--bs-warning-border-subtle`
- `--bs-danger`, `--bs-danger-text-emphasis`, `--bs-danger-bg-subtle`, `--bs-danger-border-subtle`
- `--bs-info`, `--bs-info-text-emphasis`, `--bs-info-bg-subtle`, `--bs-info-border-subtle`

The only permitted exceptions to token-only colours are:

- `color: #fff` for white text on solid `--bs-primary` backgrounds
- `background: rgba(255,255,255,.18)` and `border: 1px solid rgba(255,255,255,.4)` for the
  ghost button that sits on the primary-coloured header bar

No other hardcoded hex values are permitted anywhere in the template.

## Step 1: Create rules files

Before writing any templates, create two CLAUDE.md rules files to record the UI
conventions established by this work.

### `app/templates/CLAUDE.md`

Create this file (or append to it if it already exists) with a section covering:

**Colour policy.** All colours in templates and component `<style>` blocks must use
Bootstrap 5.3 CSS custom property tokens (e.g. `var(--bs-primary)`,
`var(--bs-danger-bg-subtle)`). No hardcoded hex values except `#fff` for white text on
solid primary backgrounds, and rgba white values for ghost buttons on coloured headers.
This ensures the UI is automatically correct in both light and dark Bootstrap themes.

**Filter button pattern.** Many templates contain rows of filter/toggle buttons. The
target pattern for new and refactored templates is:

- Render as `<button>` elements, not `<a>` tags.
- Pill shape: `border-radius: 20px`, `font-size: 12px`, `padding: 3px 11px`.
- Inactive: `background: var(--bs-body-bg)`, `border: 1px solid var(--bs-border-color)`,
  `color: var(--bs-body-color)`. Hover darkens border to `var(--bs-secondary-color)`.
- Active: `background: var(--bs-primary)`, `color: #fff`, `border-color: var(--bs-primary)`.
- Existing templates using the old `btn-sm btn-primary` / `btn-outline-secondary` pattern
  do not need immediate conversion; convert opportunistically when a template is touched
  for another reason.

**Collapsible filter panels.** Filter panels with more than two filter rows should be
wrapped in a collapsible section with a toggle ("Hide filters" / "Show filters" + chevron
icon). Visible by default on first load.

**Toolbar button grouping.** Action toolbars should group related actions with a 1px
vertical separator (`background: var(--bs-border-color)`, `height: 22px`) between groups.
Destructive actions must use `border-color: var(--bs-danger-border-subtle)` and
`color: var(--bs-danger-text-emphasis)` to distinguish them visually. Infrequent bulk
actions should be collapsed into a dropdown rather than shown as individual buttons.

### `app/templates/convenor/CLAUDE.md`

Create this file with a section covering the student card layout pattern:

**Student card structure.** When displaying a list of students with per-student submission
detail, use a card-per-student layout rather than a DataTables table. Each card has three
layers:

1. **Card header** (`background: var(--bs-tertiary-bg)`, `border-bottom: 1px solid var(--bs-border-color)`):
   student name as `<a>` at `font-size: 14px; font-weight: 600; color: var(--bs-primary)`;
   programme and cohort as plain secondary text (`font-size: 12px; color: var(--bs-secondary-color)`)
   with `·` separators on the same line — no badge styling; status pills
   (Published/Unpublished, warnings, errors) on the same metadata line; student-level
   action buttons (Journal, History, Roles, overflow dropdown) in the card header right.
   Programme and cohort carry no badge styling — plain secondary text only.
   Only Published/Unpublished and alert states use pill styling.

2. **Period tabs** — if `config.number_submissions > 1`, render a tab bar with one tab per
   period. Active tab: `color: var(--bs-primary)`, `border-bottom: 2px solid var(--bs-primary)`.
   If a period has warnings, append a small dot (`7px`, `background: var(--bs-warning)`);
   errors get `background: var(--bs-danger)`. For single-period project classes, omit the
   tab bar entirely.

3. **Period panel** — project title + owner/change link + project page button; metric
   capsules; status strip; similarity strip if present; risk factor button if needed; roles
   grid.

**Metric capsules.** Attendance metrics (Recorded, Missing, Rate) and grade metrics
(Supervision, Report, Presentation) must be rendered as two visually distinct labelled
capsules — never a single undifferentiated row.

- Attendance capsule: `border-color: var(--bs-info-border-subtle)`,
  body `background: var(--bs-info-bg-subtle)`, label `color: var(--bs-info-text-emphasis)`.
- Grades capsule: `border-color: var(--bs-success-border-subtle)`,
  body `background: var(--bs-success-bg-subtle)`, label `color: var(--bs-success-text-emphasis)`.
- Suppress the Attendance capsule entirely when no attendance data exists for the period.
- Suppress both capsules when neither attendance nor grade data exists.

**Metric value colour coding:**

- Present, positive value: `color: var(--bs-success-text-emphasis)`
- Warning / zero recorded / non-zero missing: `color: var(--bs-danger-text-emphasis)`
- Null / not yet set: `color: var(--bs-border-color)` (renders as a dim `—`)

**Status strip chips:**

- LLM result present: `background: var(--bs-success-bg-subtle)`, `color: var(--bs-success-text-emphasis)`
- LLM pending: `background: var(--bs-warning-bg-subtle)`, `color: var(--bs-warning-text-emphasis)`
- No LLM analysis: `background: var(--bs-secondary-bg)`, `color: var(--bs-secondary-color)`
- Flag ok: success subtle pair
- Flag warning: warning subtle pair with warning border
- Flag danger: danger subtle pair with danger border

**Similarity and risk strips:**

- Similarity strip: `background: var(--bs-warning-bg-subtle)`, `border: 1px solid var(--bs-warning-border-subtle)`
- Risk factor button: `background: var(--bs-danger-bg-subtle)`, `color: var(--bs-danger-text-emphasis)`,
  `border: 1px solid var(--bs-danger-border-subtle)`

**Card list vs DataTables.** The convenor submitters view does not use DataTables. Filtering
is server-side via query parameters; pagination via SQLAlchemy `.paginate()`. Other views
in the convenor dashboard that are genuinely tabular should retain DataTables.

## Step 2: Create the new template

Create `app/templates/convenor/dashboard/submitters_v2.html` alongside the existing
template. Do not modify the existing `submitters.html`.

The new template should:

- Extend the same base as the current template.
- Accept the same template variables currently passed by `submitters()`. Do not change the
  route or its context variables.
- Render filter controls (cohort, programme, state, display) as a collapsible pill-button
  panel per the conventions above, using URL-parameter links to the same
  `convenor.submitters` endpoint.
- Render the action toolbar per the conventions above, with Pipeline actions in a dropdown.
- Render one student card per submitter, with tabbed periods for multi-period project
  classes, following the card structure in `docs/mockups/submitters_cards_mockup.html`
  exactly.
- Implement tab switching in vanilla JS (no jQuery). Tabs must work without a page reload.
- Implement the filter panel collapse toggle in vanilla JS.
- Render pagination controls at the bottom of the card list.
- Preserve the LLM polling behaviour from the current template: poll
  `/documents/analysis_status/<sid>` every 12 seconds for any period panel showing an
  in-progress analysis indicator, and reload the affected card (or the full page) when
  status changes from `in_progress`.

No new CSS framework dependencies. Bootstrap 5.3 and Font Awesome 6 (both already loaded)
are sufficient. Component-specific styles go in a `<style>` block in the template for now.

All colours in the `<style>` block must follow the colour policy: Bootstrap 5.3 CSS custom
property tokens only, with the two permitted exceptions for the primary header.

## Step 3: Add a route toggle

In `app/convenor/submitters.py`, add a config-driven toggle:

```python
template = 'convenor/dashboard/submitters_v2.html'
    if current_app.config.get('SUBMITTERS_V2', False)
    else 'convenor/dashboard/submitters.html'
return render_template(template, **context)
```

Do not change any other logic in the route. The context dictionary must be identical for
both templates.
